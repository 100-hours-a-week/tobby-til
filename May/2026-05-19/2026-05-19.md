# Spring MVC

## Servlet

`DEF`

자바로 웹페이지를 동적으로 생성하고, HTTP 요청을 처리하며 응답을 생성하는 기술 표준

`BACKGROUND PROBLEM`

HTTP Message의 내용을 직접 작성하고 파싱해야했음(비효율적)

따라서 HTTP 요청, 응답을 만들어내는 Servlet이 탄생 헤더 분석, 파라미터 파싱 등의 HTTP 요청/응답 내용보다 비즈니스 로직에 집중하게 됨

`WHY`

개발자들이 비즈니스 로직과 무관한 HTTP 요청 처리 및 응답 생성 과정을 위임하기 위해 사용

## Dispatcher Servlet

`DEF`

HTTP 요청을 받아 공통 작업을 처리하고 적절한 컨트롤러(Handler)로 요청을 위임하는 요청 처리 흐름 제어 컴포넌트

- 기존 Servlet이 한 곳에서 흐름을 제어하며 발생한 문제를 해결하기 위해 Hanlder 검색, Hanlder 호출, View 생성으로 기능을 나누어 흐름을 제어하는 중심 컴포넌트가 되었음

`BACKGROUND PROBLEM`

Servlet은 GET /users GET /posts 등의 공통 작업에 **공통 로직(doGet())이 반복**되는 문제 유저의 요청 종류마다 Servlet 코드를 반복해서 적어야함

login servlet에 요청 처리(파라미터 등) / 비즈니스 로직 부르고 / 응답 전송 logout servlet 요청 처리(파라미터 등) / 비즈니스 로직 호출 / 응답 전송 …

`WHY`

공통 작업의 일원화와 중복 코드 제거

- 수십 개의 URI마다 각각 전용 서블릿을 만들던 중복 작업을 요청 최초에 Dispatcher Servlet 하나가 전담하여 한 곳에서 공통 작업 처리 (Front controller pattern)
- e.g. 고속도로 휴게소에서 주문만 받는 직원

> 공통 로직
> 
> - 요청을 처리할 수 있는지 확인 → Handler Mapping
>  - 비즈니스 로직을 수행할 수 있는 Hanlder에 요청을 전달 → Handler Adapter
> 	 - Controller는 비즈니스 로직이 있는 Model로 요청 수행
>  - 고객이 요청한 결과물을 반환 → View Resolver
> 	 - 웹 페이지(HTML), 로그인 성공 여부 등

### Dispatcher Servlet 구성 요소

공통 요청을 모두 처리하는 것이 dispatcher servlet에게 무겁기 때문에 기능을 세 곳에 나누고 중앙에서 흐름을 제어함

1. Handler Mapping
    
    - HTTP 요청 URI와 Method를 확인해 요청을 처리할 핸들러를 찾는다 e.g. /users 요청이 들어오면 해당 핸들러 호출
    - `@RequestMapping`이나 `@GetMapping` 같은 애노테이션 정보를 런타임에 탐색하여 어떤 자바 메서드가 실행되어야 하는지 동적으로 찾아내는 맵(Map) 역할을 수행
2. Handler Adapter
    
    `DEF`
    
    Handler Mapping이 찾아낸 컨틀롤러의 메서드를 Spring Framework가 호출할 수 있도록 인터페이스 형식을 맞추는 컴포넌트
    
    `WHY`
    
    컨트롤러 종류가 여러 가지이므로 이 다양성을 추상화하기 위해 인터페이스 형식의 Adapter가 필요한 것
    
    - e.g. 애너테이션 기반, **HttpRequestHandler기반 등**
3. View Resolver
    
    - 응답을 View로 생성

## Spring MVC

`DEF`

웹 요청을 최초로 받아 적절한 Controller에 분배하여 처리 흐름을 제어하고(front controller pattern), 비즈니스 로직인 Model과 View를 연결해 응답을 생성하도록 설계하는 MVC 패턴으로 구축된 Spring 기반 자바 웹 어플리케이션 프레임워크

`WHY`

- HTTP 요청, 응답 처리 기준과 흐름을 정하려 어플리케이션 구조를 일관적으로 설계하기 위함
- 관심사 분리: 비즈니스 로직(Model), 응답(View), 제어 로직(Controller)를 분리하여 코드의 독립성과 가독성을 높이기 위함
- 확상성: 컨트롤러 메서드가 특정 Servlet API에 종속되지 않고 POJO 형태로 동작하여 테스트 및 기능 확장이 쉽게 만들기 위함

## MVC Pattern

`DEF`

어플리케이션 구성 요소를 Model(비즈니스 로직, 데이터)-View(응답)-Controller(흐름 제어)로 코드의 책임을 분리하는 패턴

`WHY`

- 관심사의 분리를 통해 유지보수성을 높임
    
    하나의 파일 안에 HTML, 자바 비즈니스 코드가 있으면 코드 하나를 바꿀 때 다른 기능에도 영향이 갈 수 있다.
    
- 독립적 협업으로 개발 효율성 향상
    
    여러 사람과 협업하는 경우 각 역할의 사람들은 자신의 책임 영역 이외의 코드를 알 필요 없음
    
- 코드 재사용성 증가

# HTTPS

Hypertext Transfer Protocol Secure

`DEF`

보안에 취약한 일반 HTTP 프로토콜에 SSL/TLS 암호화 프로토콜을 얹어, 데이터 전송 과정을 안전하게 보호하는 통신 규약

`WHY-PROBLEM`

- 데이터 도청 및 변조 방지
    
    - HTTP는 평문 통신이므로 중간에 네트워크 패킷을 가로채면 비밀번호나 개인정보가 그대로 노출되었음. 이를 암호화하여 제3자가 읽을 수 없도록 만든다.
    
- 신뢰성 확보
    
    - 클라이언트가 접속한 사이트가 진짜인지 인증서를 통해 검증하기 위해

`SOLUTION`

데이터를 키로 암호화하여 제3자가 읽을 수 없게 만들어야 했다. 대칭키는 속도가 빠른 대신 키 전달이 어려웠으므로, 키 전달을 비대칭키로 하고, 통신은 대칭키로 암호화/복호화하는 방식을 사용하게 된다.

## 대칭키와 비대칭키

###  대칭키

Symmetric Key

`DEF`

데이터를 암호화할 때 사용하는 키와, 암호문을 다시 원래 데이터로 복호화할 때 사용하는 키가 동일한(하나인) 암호화 방식

![[Pasted image 20260522010350.png]]

`PROS & CONS`

장점: 암호화 연산이 매우 빠름

- 대량의 웹데이터를 실시간으로 주고받을 때 빠른 속도를 위해 대칭키를 사용

단점: 키가 탈취되면 보안이 완전히 무너지는 단점이 있어서 키 전달이 어려움

### 비대칭키 / 공개키

Asymmetric Key / Public Key

`DEF`

암호화할 때 쓰는 공개키와 복호화할 때 쓰는 개인키가 한 쌍으로 이루어져 서로 다른 암호화 방식

`WHY`

- 안전한 키 교환
    - 대칭키의 치명적인 단점 ‘키를 안전하게 전달하기 어려운 문제’를 해결하기 위해, 비대칭키를 이용한다. 대칭키를 비대칭키의 공개키로 암호화하고 서버의 개인키로 복호화하여 키 탈취 문제를 해결한다.
- 인증을 위해 (전자 서명)
    - 데이터 탈취를 공개키 암호화 → 개인키 복호화로 막은 것을 반대로 이용하여 데이터를 개인키로 암호화하고 누구나 공개키로 복호화하여 해당 데이터가 특정인이 암호화했음을 인증할 수 있음(전자 서명)

`PROS & CONS`

- 장점: 키 교환이 안전함 (공개키로 암호화하면 개인키를 갖고 있는 서버만 복호화 가능)
- 단점: 복잡한 수학적 알고리즘으로 인해 연산이 대칭키에 비해 수백배 느림 (실시간 통신 사용 불가)

## HTTPS 통신 과정

### 서버가 인증(CA)기관을 통해 인증서를 생성

![[Pasted image 20260522010748.png]]

1. 서버가 서버의 개인키-공개키 쌍을 생성
2. 서버가 도메인에 대한 주인을 인증하기 위해, 인증기관에 서버의 공개키와 정보를 전달
3. 인증기관은 서버가 도메인의 주인임을 알리는 인증서를 발급, 이때 인증기관의 전자서명(유일한 개인키로 암호화)

#### 인증서

`DEF`

인증기관이 서버가 도메인의 주인임을 인증하는 것

`구성`

- 서버의 공개키
- 인증기관의 전자서명(CA기관의 개인키로 암호화)

### TLS HandShake - 클라이언트와 서버 간 통신 세션 실행 프로세스

`DEF`

TLS 암호화를 사용하는 통신 세션 실행 프로세스, 통신 양측의 메세지를 교환해 서로를 인식하고 서로를 검증하며 사용할 알고리즘을 구성하고 세션 키에 합의한다. HTTPS 작동 원리의 근간이다.

‘인증서’를 통해 서버의 진위 여부를 확인하고, (인증서 내의) 서버 공개키를 사용해 대칭키를 암호화하여 전달한다

[자세한 내용](https://www.cloudflare.com/ko-kr/learning/ssl/what-happens-in-a-tls-handshake/)

`WHY`

- 대칭키는 전달이 어렵지만 데이터 전달이 빠르므로 통신에 사용하고 싶음
- 비대칭키는 전달은 쉬우나 처리 속도가 느림

대칭키의 통신이 빠르지만 키 탈취 시 보안이 무너지는 특징을 해결하기 위함

**TLS handshake 내의 키 교환 방식**

- 목적: 서로 세션 키를 어떻게 안전하게 나누어 가질 것인가
- 종류: RSA, DHE, ECDHE

#### **TLS Handshake 과정 - RSA 키 교환 방식**

RSA (Rivest-Shamir-Adleman) 키 교환, 세션 키 생성에 서버의 공개키/개인키가 관여하는 방식

1. Client Hello 메세지
    - 클라이언트는 서버에 접속을 시도함
    - 전달 데이터: 클라이언트 생성 난수, 클라이언트 지원 암호화 알고리즘 목록
2. Server Hello 메세지
    - 서버 응답과 인증서 전달
    - 전달 데이터: 서버 난수, 선택된 알고리즘, 인증서
        - 인증서: CA의 전자서명 + 서버의 공개키
3. Certificatate - Client의 인증서 검증
    - Client 브라우저 내 CA 공개키로 서버가 준 인증서의 CA 전자서명 복호화 → 성공하면 이 서버는 진짜라고 신뢰
4. 예비 마스터 암호(premaster secret) 전달
    - 예비 마스터 암호 생성: Client가 46바이트 난수 + 2바이트 TLS 정보로 구성된 예비 마스터 암호 생성
    - 키 교환: 예비 마스터 암호를 인증서 내 서버 공개키로 암호화하여 서버에 전송(서버의 개인키로만 복호화 가능하여 키 탈취 불가능)
5. 세션 키(master secret) 생성 (대칭키 생성)
    - 서버는 예비 마스터 암호를 서버의 개인키로 복호화
    - 서버와 클라이언트 모두 서로의 난수와 예비 마스터키를 사용해 세션 키 생성
    - 서버와 클라이언트 모두 생성한 세션 키가 같아야 함
6. 클라이언트 & 서버 준비 완료
    - 각각 세션 키로 암호화된 “finished” 메시지를 송신
7. handshake 완료, 세션 키를 이용한 통신 진행

> **RSA 방식이 위험한 이유 - 과거의 비밀을 미래의 해킹으로 풀 수 있기 때문**
> 
> 해커가 암호화된 통신을 가로채도 당장 세션 키가 없어서 복호화할 수는 없지만, 미래의 어느 시점에 서버의 개인키가 유출된다면 해커는 모아두었던 과거의 Handshake 기록 중 '서버 공개키로 암호화되었던 예비 마스터 암호'를 전부 복호화 가능
> 
> 예비 마스터 암호를 알면 과거의 세션 키도 모두 알아낼 수 있으므로, 과거의 모든 통신 데이터가 전부 해독되는 치명적인 문제가 발생

#### SSL / TLS

**SSL(Secure Sockets Layer)**

데이터 보안 전송을 위한 암호화 프로토콜, 보안 취약점이 존재하여 더 이상 사용하지 않음

**TLS(Transport Layer Security)**

SSL의 개선 버전

#### TLS handshake 내 키 교환 방식 종류

- RSA (Rivest-Shamir-Adleman) 키 교환
    - **정의**: 클라이언트가 생성한 예비 마스터 암호를 서버의 공개키로 암호화하여 전달하고, 서버가 자신의 개인키로 복호화해 대칭키(세션 키)의 재료로 사용하는 전통적인 방식
        - 클라이언트가 예비 마스터 암호 생성 → “서버 공개키로 암호화” → 서버가 개인키로 복호화하여 세션키(대칭키)로 사용
    - 치명적 단점: 완전 순방향 비밀성(PFS, Perfect Forward Secrecy) 미지원
        - 과거의 비밀이 미래의 해킹에 의해 뚫린다
        - 해커가 현재의 암호화된 통신 데이터를 모두 수집해 두었다가, 미래에 서버의 고정 개인키를 탈취하게 되면, 과거에 수집해 둔 모든 통신 기록의 예비 마스터 암호를 복호화할 수 있음
        - TLS 1.3에서 RSA가 퇴출된 이유
- DHE (Diffie-Hellman Ephemeral) 키 교환
    - **정의**: 클라이언트와 서버가 네트워크를 통해 키 자체를 주고받지 않고, 공개된 파라미터만 교환한 뒤 서로 독립적인 수학적 계산을 통해 동일한 대칭키(세션 키)를 도출해내는 방식
    - **개인키의 용도 변화: 대칭키 생성에 관여하지 않고 전자 서명에만 사용**
        - 서버의 고정된 개인키는 더 이상 대칭키 생성(암호화)에 개입하지 않고, 파라미터를 보낼 때 '이 파라미터는 진짜 서버가 보낸 것이 맞다'는 것을 증명하는 **전자 서명** 용도로만 사용
    - 사용 이유: 완전 순방향 비밀성(PFS) 달성
        - Ephemeral(임시적인)이 의미하듯, 세션을 생성할 때마다 매번 새로운 일회용 난수(임시 키 쌍)를 만들어 대칭키를 도출
        - 따라서 서버의 고정 개인키가 유출되더라도, 이미 통신이 끝나고 폐기된 과거 세션의 대칭키는 절대 알아낼 수 없음
- ECDHE (Elliptic Curve Diffie-Hellman Ephemeral) 키 교환
    - **정의**: DHE 방식에 타원 곡선 암호학(ECC, Elliptic Curve Cryptography)을 결합한 진화된 키 교환 방식
    - **사용 이유**: **최적의 보안성과 압도적인 연산 속도 확보**
        - 일반적인 DHE 방식은 소수(Prime) 연산을 사용하기 때문에 CPU 연산 부하가 크고 느리지만, ECDHE는 타원 곡선 수학을 적용하여, 기존보다 훨씬 짧은 키 길이로도 동일한 수준의 보안(PFS 포함)을 제공
        - 서버의 부하를 획기적으로 줄이면서 처리 속도를 비약적으로 끌어올렸기 때문에, 현재 TLS 1.2, TLS 1.3 사실상 표준
