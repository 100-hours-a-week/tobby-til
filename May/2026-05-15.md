## 상속

`DEF`: 자식 클래스가 부모 클래스의 필드/메서드를 물려받는 것

`WHY`: 코드 재사용, 공통 기능을 갖는 클래스를 상속해 유지보수 용이

**사용 방법**

`class 자식클래스명 extends 부모클래스명`

**상속을 결정하는 단어**

`IS`

두 관계를 IS로 표현할 수 있다면 상속 관계로 표현할 수 있다.

e.g. 검은 무기다, 자동차는 탈 것이다. 

**상속 관계가 될 수 없는 단어**

`HAS`

HAS 관계는 상속 관계로 표현할 수 없다.

e.g. 무기는 강화치가 있다, 

### 분류/조합

1. 클래스 관계가 분류면 상속

    - 무기 - 검, 무기 - 활 등 개념 나뉘어서 "분류"가 필요하면 상속 관계

2. 클래스 관계가 조합이라면 필드, 속성

    - 아이스 아메리카노와 아메리카노의 관계는 "얼음" 속성 유무 => 아아와 아메리카노는 별개의 개체가 아니다

## Polymorphism 다형성

`DEF`: 정보의 다양한 형태

- e.g. Car 클래스를 상속받은 가솔린차와 전기차의 drive 메소드가 다른 결과를 출력함

### Method Overloading

`DEF`: 같은 이름의 메서드를 매개변수 개수 / 타입을 다르게 정의

`WHY`: 같은 이름의 메서드를 다른 형태로 정의하며 코드 일관성, 확장성, 유지보수성을 높이기 위해

- e.g. 클래스 생성자의 매개변수에 따른 변경

### Method Overriding

`DEF`: 부모 클래스의 메서드를 자식 클래스에서 재정의하여 동작을 바꾸는 개념(메서드 이름, 매개변수, 리턴 타입 유지)

`WHY`: 상속받은 메서드를 자식 클래스에 맞게 바꿔 코드의 유연성/확장성을 높이기 위해

- e.g. Car 클래스의 drive 메서드를 자식 클래스인 Bus에서 일정 속력 이상인 경우 속력이 제한된 상태로 달리도록 메서드 Overriding

## Encapsulation 캡슐화

`DEF`: 외부에서 객체 내부에 접근하지 못하도록 감추고, 메서드를 통해 간접적으로 접근하도록 하는 객체지향 개념

`WHY`: 객체 내부의 데이터를 보호하고, 메서드에서 유효성 검사를 하며, 무분별할 접근을 막기 위해

- e.g. 캡슐화를 하지 않으면 은행계좌의 잔액을 음수로 직접 바꾸는 경우 `bank.balance = -1`처럼 데이터를 마음대로 수정할 수 있다.

```java
public class Bank {
    private int balance;
    public void setBalance(int b) {
        if (b < 0) {
            sout("잔고는 음수가 될 수 없습니다.");
            return;
        } else {
            this.balance = b;
        }
    }
}
```

## Interface

`DEF`: 세부 기능 구현 없이 클래스가 구현해야 할 메서드의 집합만을 정의한 설계 명세서

`WHY`: 설계 단계에서 구조를 먼저 정의할 수 있도록하여 설계와 구현을 분리하기 위해

### Interface vs Inheritacne

| 구분 | Inheritance | Interface |
|---|---|---|
| 관계 | IS-A (~는~이다) | CAN DO |
| 예시 | 탈 것 - 자동차 | 수영가능, 운전가능 |
| 특징 | 하나의 부모 | 중복 구현 가능 |

장점

- 모든 기능을 메서드로 넣을 수 없는데 (나중에 추가되기도 하므로) 인터페이스는 중복해서 부여할 수 있으므로 확장성을 보장한다(ex. 비행 가능한 자동차)

## Generic <T>

`DEF`: 데이터 타입을 지정하기 않고 클래스/메서드가 사용할 때 지정하는 매개변수 문법

`WHY`: 기존에는 모든 타입을 받을 수 있도록 사용한 Object 객체를 사용했으나 사용할 때와 저장할 때 형변환이 필요했다. 

사용처

- 공통 API 응답 규격

- Collection 프레임 워크: List<T>, Map<K, V>, Set

### Bounded Type 제한된 타입

`PROBLEM`: 자유로운 제네릭은 위험함

`SOLUTION`: `<T extends 부모클래스>`로 T에는 부모클래스/자식 클래스만 가능

## Collection

`DEF`: 데이터 집합을 다루기 위한 자바 표준 자료구조 **인터페이스**

`WHY`: 왜 아래 세 자료구조는 Collection 인터페이스를 구현한 클래스인가? 

**List**

- 데이터 순서 유지, 중복 허용

**Set**

- 중복 없는 집합
- 순서를 보장하지 않음(넣은 순서 유지 안 됨)
- contains 연산이 빠름

**Map**

- Key-Value 구조
- Key
    - 유일한 값
    - 검색이 빠름
- Value
    - 중복 가능


## Thread

`DEF`: 한 프로세스 내에서 동시에 실행할 수 있는 독립적인 실행 흐름 단위

- 싱글 스레드: 하나의 스레드
- 멀티 스레드: 한 프로세스 내 다중 스레드

`WHY`: 여러 작업을 사러 영향을 주지 않고 동시에 실행하기 위해

- e.g. 대용량 데이터 배치 작업 시, 서버가 멈추기 않게 만들기 위함
- 스레드를 실행하면 기존 흐름(main)과 독립적으로 실행됨

**주의사항**

1. try-catch 사용하기: 스레드 내부의 에러 발생 시 메인 프로세스에서 알 수 없음
2. 여러 스레드에서 같은 값을 보고 있을 때, 수정 반영이 느릴 수 있음

### 사용 방법

1. Thread 클래스 상속 받기
2. Runnable 인터페이스 구현 객체를 Thread에 전달해 실행

### Runnable

- 스레드에서 해야하는 일을 설계하는 인터페이스
- Java의 상속은 한 클래스만 가능하기 때문에 Runnable을 사용

**실무에서 멀티 스레드 사용 방법**

- new Thread를 하지 않음
- ExecutorService 사용 (작업 단위 Runnable)

## ExecutorService

`PROBLEM`: new Thread의 위험성이 있음

1. OS가 관리하는 무거운 자원
2. 리소스 고갈 문제(OOM): 작업마다 스레드를 만들때, 해당 작업이 한 순간에 너무 많아지면 문제가 발생
3. 스레드 관리의 어려움: 명시적으로 종료해줘야함, 에러를 처리하기 힘듬

`SOLUTION`: Thread Pool을 만들어두고 Thread를 사용

**구성**

1. Thread Pool: Thread를 생성하고 관리하는 곳, 사용 후 이곳에 Thread 반환
2. Blocking Queues: 제출된 작업들이 Thread를 할당받기를 기다람
3. Task Submitters: 작업 제출

`WHY`

1. 자원 보호
2. 성능 향상
3. 관리 용이

**현업에서 ThreadPool**

- `@Async`를 사용해 내부적으로 executorService를 작동

## Synchronized

`DEF`: 여러 스레드가 동시에 공유 자원에 접근하지 못하게 해당 자원의 접근을 하나로 임시 제한하는 키워드

`WHY`: 공유 자원을 동시에 접근하면 충돌이 발생하기 때문

- e.g. 공유자원 balance 잔고 100원일때, 동시에 100원씩 인출하여 총 200원을 인출하려고 시도

### Critical Section 임계 영역

`DEF`: 공유 자원에 접근하는 코드 일부분, 여러 스레드가 동시에 실행하면 안됨(충돌 발생)

## Lock

`DEF`: 여러 스레드가 한 공유 자원에 접근하지 못하도록 막는 장치, 스레드가 임계 영역에 진입할 때 Lock을 획득하고 다른 스레드는 그 영역에 접근하지 못함

`PROBLEM`: Synchronized의 문제

- 먼저 대기하고 있는 thread를 알 수 없음
- 무한히 대기할 수 있음

`SOLUTION`: 먼저 온 thread를 먼저 처리하기 위해 Lock 등장

**Lock의 특징**

- 대기 조건을 정할 수 있음
- 무한 대기를 안 할 수 있음(대기 시간을 정할 수 있음)

**사용 방법**

`lock.lock()`으로 임계 구역 접근 시 락 획득, 작업 수행 후 finally 블록에서 lock.unlock()으로 락 해제
