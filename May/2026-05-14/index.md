# TIL
## 절차 지향 / 객체 지향
**절차 지향 프로그래밍**

DEF: 순서대로 실행되도록 코드를 작성하는 것

PROBLEM: 여러 곳에 같은 내용이 있을 때 코드 수정이 힘들다(하나하나 찾아서 수정해야함)

**객체 지향 프로그래밍**

DEF: 데이터와 메서드를 객체로 묶어 프로그램을 설계하는 것

특징

1. 캡슐화(Encapsulation)
2. 추상화(Abstraction)
3. 상속(Inheritance)
4. 다형성(Polymorphism)

**상속**

DEF: 자손 클래스가 부모 클래스의 데이터를 복제하는 것

## Class

DEF: 속성과 메서드를 묶어 객체를 생성하기 위한 템플릿

WHY: 코드 중복을 줄이고 재사용성을 높이기 위해, 속성과 동작을 하나로 묶어 객체 단위로 모듈화

## Git

PROBLEM: 코드를 여러 곳에 저장하는 경우 코드의 버전 관리가 어려움(따라서 여러 사람과 협업 시 동시에 하나의 프로젝트를 수정하기 힘들다)


프로그램이 몇 번 수정되었는지 관리하는 버전 관리 시스템을 만들어 이를 해결했음

### VCS 버전 관리 시스템

프로그램이 몇 번 수정되었는지 관리하는 시스템

#### 중앙 집중식 VCS

서버에서만 프로그램의 버전을 관리함

PROS & CONS

- PROS
    - 공유/관리 용이
- CONS
    - 중앙 시스템에 문제가 생기면 데이터가 모두 유실될 수 있음

#### 분산 VCS

PROS & CONS

- PROS
    - 서버의 데이터가 유실되어도 복구 가능성이 있음
- CONS
    - 동기화 작업이 필요해 복잡하다

### Git 작동 방식

#### local git

1. `git init`으로 작업 공간을 git이 추적하도록 한다.
2. 작업 공간에서 수정한 파일을 `git add`하여 stage 상태로 변경
3. `git commit`으로 stage 상태의 파일을 history로 이동
4. `git push`로 원격 저장소로 변경 사항을 저장

#### git 명령어

- `git remote`: 원격 저장소 연결 관리
- `git status`: 현재 저장소 상태 확인
- `git pull`: 원격 저장소 레포지토리의 최신 내용을 로컬 저장소로 가져옴
    - `git fetch` + `git merge` 기능을 합한 것
- `git show 커밋아이디`: 커밋 정보 조회
- `git blame`: 수정 내역 확인
